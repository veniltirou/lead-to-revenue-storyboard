The Friday Debrief (OIC)

A 10-scene follow-up to "Oracle Integration Cloud." Harrison built the integration and James won the deal — now James needs to be able to explain how to people who won't read a line of JSON. This chapter follows Harrison walking James through the real build, entirely in plain language, using the actual payloads only as visual backup.

Business Scenario: This chapter assumes the build in "Oracle Integration Cloud" already happened — Home, Sweet Home's orders are flowing into ABC Corporation's Oracle Order Management automatically. James now needs to carry that story to his own stakeholders without touching code himself, so he asks Harrison for a Friday afternoon walkthrough: no code, just what actually happens, in order. It closes on the question that matters most to a sales rep — will this same approach work for the next non-Oracle customer too?

👥 Characters in This Chapter

Harrison — CS undergrad, rising junior, summer intern at ABC Corporation, explaining his own build back in business terms
James (GREEN shirt) — ABC Corporation Sales Rep, translating the technical win for his own stakeholders
A note on technical accuracy: The REST field names shown in this chapter (BuyingPartyNumber, ProductNumber, FreezePriceFlag, SourceTransactionSystem, TransactionalCurrencyCode, BusinessUnitId, OrderNumber, HeaderId, StatusCode) are checked against Oracle's published salesOrdersForOrderHub REST API schema for Oracle Fusion Cloud Order Management. The on-screen monitor mockups themselves are illustrative reconstructions, not actual product screenshots — Oracle's UI and API details can change across releases, so treat this chapter as educational rather than a substitute for current Oracle documentation.

Scene 1 — The Assignment

Explain It Friday, No Code

What this shows: James credits Harrison's Oracle Integration Cloud work for winning the Home, Sweet Home deal, then asks for a plain-language walkthrough — no JSON — that he can repeat to his own stakeholders. They set Friday afternoon to reconvene, no code, just what actually happens, in order.

Scene 2 — The Service Window

One Request In, One Receipt Back

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: Harrison opens with the actual payload Home, Sweet Home's system sends — a simple POST request to ABC's REST endpoint. He frames it in plain terms: think of it as a dedicated service window. HSH pulls up, hands over the order, and gets a receipt back immediately, same stop — one HTTP call, not a back-and-forth.

Scene 3 — Mapper 1, the Concept

Why a Mapper Exists

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: James asks why a mapper is even needed. Harrison lays out the shape of it: OIC receives HSH's simple request, maps and fills in whatever additional detail ABC's Oracle Order Management needs, then sends the fully detailed request onward. This is the concept, before the specifics.

Scene 4 — Mapper 1, the Specifics

Lookups, Fixed Values, and What Each One Means

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: The actual mapping table: propertyId and equipmentRequested are resolved through live lookups into BuyingPartyNumber and ProductNumber, while BusinessUnitId, TransactionalCurrencyCode, SourceTransactionSystem, and FreezePriceFlag are fixed values Order Management needs but HSH's system never sends. Harrison confirms it's now ready to POST.

Scene 5 — Invoking Order Management

Oracle Order Management Creates the Sales Order

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: OIC's Invoke step calls Oracle Order Management with the mapped payload. Order Management creates the sales order and responds with a 200 OK — the order number, its internal header ID, and a Processing status — confirming the order landed on Oracle's side.

Scene 6 — Mapper 2, the Response

Translating the Response Back — On Purpose, Not Completely

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: Same translation, opposite direction. Harrison deliberately keeps only what HSH actually cares about — the order number and its status — and leaves out internal Oracle identifiers like HeaderId and Revision that mean nothing outside ABC's system.

Scene 7 — Responding to HSH

Simple In, Simple Out

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: The response HSH's system actually receives — a clean, minimal JSON confirmation with a status, both order numbers for cross-reference, and a plain message. James's takeaway: all the complexity stays on ABC's side of the window.

Scene 8 — The Logger

Why Log When OIC Already Tracks Every Run

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: James asks why a Logger is even necessary. Harrison's answer: OIC's built-in run history confirms an order passed through, but a Logger records exactly what was received and what was sent — the detail actually needed to debug a problem, rather than just knowing one occurred.

Scene 9 — The Whole Flow, in One Breath

Seven Steps, Start to Finish

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: Harrison closes the technical walkthrough with the full sequence in plain language: receive, log, map, POST, log the response, map the response down to the essentials, respond. James recognizes it as the complete mechanism, in one breath.

Scene 10 — Same Pattern, Every Customer

Does This Repeat for the Next Deal?

📱 On mobile? Pinch to zoom to read the details clearly.

What this shows: Wrapping up for the week, James asks the real payoff question: will the same approach work for the next customer running their own non-Oracle system, like the automobile-sector prospect already in the pipeline? Harrison confirms it — this isn't a one-off, it's a repeatable pattern for any customer running third-party, non-Oracle SaaS.
